# Production Deployment Blueprint

This document defines the deployment blueprint for FlowPilot AI, mapping services, DNS domains, environments, rollbacks, and verification checklists.

---

## 1. Production Stack Diagram

```
[ DNS Routing: Cloudflare ]
       │
       ├──> Frontend (Vercel) ───────────> app.flowpilot.ai
       │
       └──> Backend (Railway) ───────────> api.flowpilot.ai
               │
               ├──> DB: Supabase Postgres
               ├──> Queue Store: Redis
               └──> Monitoring: Sentry & PostHog
```

---

## 2. Infrastructure Provider Evaluation

### 2.1 Web Server: Railway vs. Render
- **Recommendation:** **Railway** (Winner)
- **Why:** 
  - Railway does not force a "cold start" sleep on inactive containers, preventing the HTTP 30-second delay web visitors experience on Render free-tier APIs.
  - Better native Redis deployments.
  - Faster builds and cheaper bandwidth pricing.

### 2.2 Datastore: Supabase Postgres
- Managed Postgres instance.
- Connection pooling natively enabled.

---

## 3. Domains & SSL Configuration

- **Frontend Target:** `app.flowpilot.ai` (DNS CNAME pointing to Vercel `cname.vercel-dns.com`).
- **Backend API Target:** `api.flowpilot.ai` (DNS CNAME pointing to Railway service endpoint).
- **SSL Certificates:** Automatic Let's Encrypt certificates generated by Vercel and Railway.
- **Proxy Management:** Cloudflare DNS handles proxy masking, SSL termination, and basic DDoS protections.

## 5. Deployment Sequence

1. **Deploy Supabase Postgres Database:** Create tables and apply SQL migrations.
2. **Deploy Redis Cache:** Provision Redis instance on Railway.
3. **Deploy Backend API on Railway:** Wire environment parameters and link database.
4. **Configure DNS Records:** Route `api.flowpilot.ai` and `app.flowpilot.ai` DNS entries in Cloudflare.
5. **Deploy Frontend on Vercel:** Wire API URLs and build settings.

---

## 6. Backups & Rollback Strategies

### 6.1 Database Backups
- **Frequency:** Daily automated snapshots in Supabase (stored for 7 days).
- **Point-In-Time Recovery (PITR):** Enabled on Supabase Pro plan for transaction-level rollbacks.

### 6.2 Rollback Strategy
- **Frontend Vercel:** Revert deployments instantly via Vercel Dashboard in 1-click.
- **Backend Railway:** Deploy previous Docker build image tagging in 1-click.
- **Database Rollbacks:** Run rollback migrations using custom down-scripts (e.g. `npm run migrate:down`).
